아래 프롬프트는 type 해서 옮긴 프롬프트로 일부 오탈자가 있을수있음 애매한 부분은 적절히 판단해서 상황에 맞게 조절.

#FLASK 기존 서버에 report 분석 모듈 추가 : SQLiTE + S3 drive 저장 구조 구현

너는 기존 FLASK 사내 웹 프로젝트에 새 report 분석 모듈을 붙이는 시니어 백앤드 엔지니어다.
기존 기능은 건드리지말고 '/pe/report' 경로 아래에만 새 기능을 추가해라.

현재 HOST:
- FLASK 사용
- SQLITE 사용
- 기존 DB : DB/TESTSPEC/test_spec.db (신규로는 DB/PE/report/report.db 이렇게 하면되지않을까)
- 기존 route : 'pe/report'
- templates 폴더 공유

새 report module:
- Raw CSV 입력 
- Summary 입력
- Yield 분석
- CPK 분석
- FAIL ITEM 분석
- Plotly 산포도 JSON 생성
- SQLite summary 저장
- Plotly JSON S3 drive 에 저장
- 같은 요청 + 옵션 재계산 방지

중요 : 
-모든 새 router 는 '/pe/report' prefix
-모든 새 table 은 'report_' prefix
- 기존 프로젝트 최소 수정
- RAW CSV 와 Plotly Full JSON 은 SQLite 저장 금지 local analysis 가 아니라 S3 서버에 저장

--- 

## 1. 핵심 저장 원칙

데이터를 3개로 분리한다.

1) Raw 데이터
- CSV 파일
- 파일 시스템 저장(서버 디스크 저장 고려중)
- SQLite 저장 금지

2) Summary 데이터
- summary, yield 의 모든 정보, cpk 의 모든 정보, fail_item의 모든 정보 
- SQLite 저장

3) plotly JSON
- 크므로 DB 저장 금지
- S3 서버에 object로 저장
- SQLite 에는 S3 object key/path 만 저장

원칙:
파일은 원본, DB는 요약, Plotly 는 S3 object storage

---

## 2. session_id와 analysis_key

session_id = 사용자 실행 기록
analysis_key = 분석 결과 재사용 기준

같은 파일 내용 + 같은 옵션이면 session_id가 달라도 같은 analysis_key를 사용한다.
분석 결과는 session_id가 아니라 analysis_key 기준으로 재사용 한다.

---
## 3. analysis_key 생성

contect_hash = 파일 내용 기반 sha256
options_json = json.dumps(options, sort_keys = True, ensure_ascii = False, separators =(',',':')
analysis_key = sha256(contect_hash + ":" + options_json)

주의:
-파일명 기반 금지
- 파일내용 기반 해시 사용
- 파일 해시는 streaming 방식으로 계산
- 전체 파일을 한번에 read 금지

## 4. SQLite DB

새 DB 는 가능하면 별도 파일 사용:

DB/pe/report/report_session.db

DB 초기화 시 PRAGMA 적용
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA temp_store = MEMORY;
PRAGMA  busy_timeout = 5000;
대량 insert/read 는 1000개 단위 chunk 처리를 고려한다.

## 5. 테이블 설계

### report_session

CREATE TABLE If NOT EXISTS report_session(
   id INTEGER PRIMARY KEY AUTOINCREMENT,
session_id TEXT UNIQUE NOT NULL,
analysis_key TEXT,
file_name TEXT NOT NULL,
file_path TEXT,
content_hash TEXT,
status TEXT DEFAULT 'pending',
created_at INTEGER,
updated_at INTEGER,
error_message TEXT
);

status:
pending, running, analysisd, done, failed

---

### report_analysis_summary

CREATE TABLE IF NOT EXISTS report_analysis_summary(
id INTEGER PRIMARY KEY AUTOINCREMENT,
analysis_key TEXT NOT NULL,
session_id TEXT,
item_name TEXT NOT NULL,
bin_number INTEGER,
yield_percent REAL,
fail_count INTEGER,
cpk_val REAL,
mean_val REAL,
stdev_val REAL,
lsl REAL,
usl REAL
unit TEXT,
created_at INTEGER,
UNIQUE(analysis_key, item_name, bin_number)
);

yield, cpk, fail_count 는 JSON 이 아니라 column 으로 저장한다.

### report_object_info

CREATE TABLE IF NOT EXISTS report_object_info (
analysis_key TEXT PRIMARY KEY,
content_hash TEXT NOT NULL,
options_json TEXT NOT NULL,
object_type TEXT NOT NULL,
s3_bucket TEXT,
s3_key TEXT NOT NULL,
s3_uri TEXT,
created_at INTERGER,
last_accessed INTEGER
);

object_type:
plotly

주의:
- plotly JSON 본문은 SQLite 에 저장하지 않는다.
- SQLite 에는 s3_key, s3_uri 	같은 위치 정보만 저장한다.

### 7. S3 저장 구조

S3 object key 예시:
pe/report/plotly/{analysis_key}.json

SQLite 에는 다음만 저장한다.
-s3_bucket
-s3_key
-s3_uri

S3 설정값은 config 로 분리한다.

REPORT_S3_ENDPOINT
REPORT_S3_BUCKET
REPORT_S3_ACCESS_KEY
REPORT_S3_SECRET_KEY
REPORT_S3_PREFIX = "pe/report/plotly"

S3 접속 방식은 boto3 호환이라고 가정하되 실제 환경에 맞제 endpoint_url 설정을 분리한다.
Plotly JSON 은 DB에 저장하지 말고 파일로만 저장한다.

---

### 8. 동시 요청 처리
같은 analysis_key 로 동시에 요청되면 중복 분석 금지
local cache 파일은 만들지 않는다.
중복 분석 방지는 SQLite 기반 lock 테이블 또는 report_session 상태를 이용한다.

간단한 방식:
analysis_key 처리 시작시 러닝 상태 기록
-같은 analysis_key 가 이미 러닝이면 잠깐 대기 후 summary와 s3 object 존재 여부를 다시 확인
-먼저 끝난 요청의 결과를 재사용

가능하면 별도 테이블을 추가한다.

## 10. /pe/report/analyze 흐름

1. 요청 수신
2. session_id 생성
3. report_session 에 pending 저장
4. 파일 존재 확인
5. content_hash streaming 계산
6. options_json 정규화
7. analysis_key 생성
8. report_session 에 analysis_key, content_hash 저장
9. report_analysis_summary 에서 analysis_key 조회
10. 있으면 analysisd=true 반환
11. 없으면 analysis_key lock 획득
12. lock 안에서 다시 캐시 조회
13. 그래도 없으면 분석 실행
14. summary 를 report_analysis_summary에 batch insert
15. plotly json 생성
16. analysis/plotly/{analysis_key}.json 저장
17. report_analysis_info 저장
18. report_session status = done
19. session_id, analysis_key, analysisd, summary 반환

---

## 11. pe/report/plot 흐름

1. analysis_key 결정
2. analysis/plotly/{analysis_key}.json 확인
3. s3 에 object 존재 여부 확인 있으면 s3 json 다운로드 후 반환
4. 없으면 lock 획득
5. lock 안에서 다시 s3 object 확인
6. 없으면 raw 파일과 options로 plotly json 생성
7. s3 에 업로드
8. report_analysis_info 업데이트
9. plotly JSON 반환

## 16. 보안/에러

file_path 는 uploads/report/ 아래 인지 검증.
../ path traversal 차단.
analysis_key는 sha256 hex 문자열인지 검증.

S3 endpoint, bucket, key 정보는 config 분리하고 코드에 하드코딩 하지말것

파일없음, csv 파싱 실패, 분석 실패시:
- report_seesion.status == Failed
- error_message 저장

s3 object 다운로드 실패
- 명백한 에러 반환
-필요시 plotly JSON 재생성 후 S3 재업로드

S3 object JSON 손상:
- 해당 object 삭제 또는 overwrite 후 재성성

SQLite lock timeout:
명백한 에러 반환

## 17. 테스트

다음을 검증하라.

1. 같은 파일 + 옵션 2회 요청시 두번째 retuse == true
2. 파일 내용 변경시 content_hash/analysis_key 변경
3. options 변경 시 analysis_key 변경
4. plotly JSON 은 SQLite나 local cache가 아니라 s3 에 저장됨
5. SQLite에는 S3 object key 만 저장됨
6. 같은 Analysis_key 동시 요청시 분석은 한번만 실행
7. session_id가 달라고 analysis_key 가 같으면 summary 재사용

## 18. 최종원칙
파일은 원본이다.
DB 는 요약과 인덱스이다
plotly 는 s3 에 저장한다.
session_id는 실행 기록이다.
analysis_key 는 재사용 기준이다.
같은 파일 내용과 같은 옵션이면 다시 계산하지 않는다.
local cache 방식은 일단 사용하지 않음
기존 Flask Project 와 충돌 없이 '/pe/report' 모듈로 붙인다.
