단계별 부하 및 예상 시간 (2000 subjects × 3 schools 기준)
#	단계	위치	부하 종류	예상 시간
1	CSV 업로드 (네트워크 전송)	양방향	네트워크 (~60MB 총합)	5-15초
2a	pandas CSV 파싱 (3 files)	서버	CPU + RAM (~250MB peak)	~3초
2b	CDF 계산 (2000 × 3 = 6000회)	서버	CPU (numpy 단일 스레드)	~15-20초
2c	JSON 직렬화 + 디스크 쓰기 (2000 파일)	서버	CPU + 디스크 IO	~10-15초
2d	cumulative.html 생성	서버	CPU 미미	<1초
3	/view 응답 + HTML 다운로드	양방향	네트워크 (~70KB)	<1초
4a	Plotly.js CDN 로드	클라이언트	네트워크 (~3MB)	1-2초 (첫 방문)<br>0초 (캐시)
4b	priming: 2000셀 SVG 사전 생성	클라이언트	CPU + GPU (WebGL) + IndexedDB IO	2-3분
4c	viewport 진입 셀 활성화	클라이언트	CPU + GPU	수십 ms/셀
5a	hover → Plotly 활성	클라이언트	CPU + GPU	~100ms
5b	zoom/pan/스크롤	클라이언트	GPU (scattergl)	<16ms (60fps)
합계
시점	시간
업로드 시작 → 페이지 표시 (단계 1~3)	~30-60초
첫 페이지 진입 → priming 완료 (4b)	추가 2-3분
두 번째 방문 (IndexedDB hit)	priming 거의 0초
핵심 부하 분포
서버: 빌드 시점에 ~30-40초 동안 한 번만 CPU/RAM 사용 (~250MB peak). 이후엔 sendfile로 메모리 0
클라이언트: priming 2-3분 동안 백그라운드 CPU/GPU. 그 후엔 거의 idle, 인터랙션 시점에만 GPU
네트워크: priming 시 2000 × ~150KB = ~300MB 전체 (localhost는 무시, WAN이면 IndexedDB hit 후 재방문은 0)
설계 의도: 서버는 가볍게 (4GB 환경 동거), 무거운 시각화는 클라이언트 GPU/IndexedDB로 떠넘김. 사용자가 첫 방문 시 2-3분 priming을 한 번만 견디면 그 후엔 휠 스크롤 무한히 부드러움.
